# Sprint 1-2: 页面动作与交互契约 (交付对接版)

## 1. 核心业务流程契约 (Sprint 2 重点)

### A. 报表异步导出流程 (17-29)
- **触发**: 用户点击 [导出异步报表] 按钮。
- **请求**: 前端发送 `GET /api/admin/consume/reports/{type}/export`。
- **响应**: 后端返回 `jobId`。
- **状态追踪**: 
  - `pending`: 排队中，系统资源调度。
  - `processing`: 生成中，正在从大数据引擎聚合数据。
  - `success`: 已就绪，提供 OSS 下载链接。
  - `failed`: 失败，需提示具体异常（如数据量过大）。
- **下载入口**: 后台“系统下载中心”或顶栏通知。

### B. 调账幂等约定 (13)
- **Header**: 调账/手动充值接口必须携带 `X-Request-ID` (UUID v4)。
- **后端逻辑**: 后端根据该 ID 在 Redis/DB 中锁定 24 小时，重复请求直接返回 `409 Conflict`。
- **前端逻辑**: 进入调账弹窗时生成 ID，点击确认后该 ID 锁定直到当前流程结束或报错。

### C. 退款状态机 (15)
- **待审 (pending)**: 记录已产生，等待财务人员操作。
- **执行中 (processing)**: 审批已通过，系统正在调用微信/支付宝退款接口。
- **成功 (success)**: 三方支付网关返回成功，资金已入用户账户。
- **失败 (failed)**: 三方支付返回失败（如余额不足），需人工重试。

### D. 设备指令状态 (31)
- **已下发 (queued)**: 指令已存入后端指令队列。
- **已执行 (executed)**: 终端设备心跳拉取后执行完毕并回传。
- **超时 (timeout)**: 60 秒内未收到设备响应。

## 2. 基础交互约定 (Sprint 1)
- **数据刷新**: 所有删除/修改操作成功后，前端应局部或全局刷新列表。
- **审批校验**: 驳回操作时，`reason` 字段为强校验必填项。
- **实时预览**: 工号规则等配置页需在 Input 修改时实时（Debounce 300ms）计算预览值。
